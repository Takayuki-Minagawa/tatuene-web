# UI再設計プラン — Excel風セルからの脱却

> 4エージェントによる並行調査（入力UI / 図面エディタ / データモデル / 出力・参照表）を管理エージェントが統合。
> 目的：「表が分かりやすい所は表、それ以外はテキストボックス等のコンパクトなUIへ」「見取り図の背景に表が透けて見えるのを解消」。

---

## 0. 結論（先に要点）

- **データ層・計算エンジン・PDF出力は一切変更不要**。すべてセル番地（A1形式）で動いており、UI層だけを差し替えても計算結果・保存ファイルの互換性は保たれる。
- アプリは**すでにハイブリッド**。表紙・部材性能・計算（現状/改修後）の各シートは `InputSheet`（フォーム）化済みで、単一項目は `FormField`（ラベル＋テキストボックス／セレクト）、密な表は `RefTable`（横スクロール表）で出している。
- 残る「Excel風セルっぽさ」と「図面背景の表透け」は、**局所的な2点の改修**で解消できる。大規模な作り直しは不要。

## 1. 現状アーキテクチャ

### 1.1 画面の出し分け（`src/app/page.tsx:40, 279-290`）
```
getSheetLayout(sheet) が定義あり → InputSheet（フォーム表示）
                       定義なし → SheetGrid（Excel忠実グリッド）
```
- フォーム化済み：**表紙 / 部材性能 / 計算シート（現状）/ 計算シート（改修後）**
- グリッドのまま：**評価シート**（＝帳票本体。PDFはこれを忠実描画するので変更不可）

### 1.2 UIコンポーネントの役割
| コンポーネント | 役割 | 種別 |
|---|---|---|
| `FormField` (`components/FormField.tsx`) | 1項目＝ラベル＋入力欄＋単位＋解説 | **テキストボックス系** |
| `FormSection` (`components/FormSection.tsx`) | 折りたたみセクション。`fields`/`reftable`/`drawing` の3種を内包 | 枠組み |
| `RefTable` (`components/RefTable.tsx`) | 複数行×複数列の密な表を横スクロールで再現（結合セル・列幅を保持） | **表形式** |
| `CellInput` (`components/CellInput.tsx`) | 個別セルの編集（番地 `addr` を受け取るだけ） | 共通部品 |
| `SheetGrid` (`components/SheetGrid.tsx`) | Excel忠実グリッド。評価シート＆PDF用 | 帳票 |
| `PlanDrawing` / `DrawingEditor` (`components/`, `components/drawing/`) | 間取り図の作図 | 図面 |

### 1.3 データ層（変更不要なことの根拠）
- 入力は `setInput(sheet, addr, value)` → HyperFormula が `{sheet,row,col}` で計算（`engine/workbook.ts`, `engine/store.ts`）。
- 保存は `'{Sheet}!{Addr}'` キー（`lib/storage.ts`, `lib/autosave.ts`）。**過去の .zip/.json と後方互換**。
- 数式はExcel原本のセル参照をそのまま使用（`engine/transform.ts` は日本語シート名→ASCIIエイリアス置換のみ）。
- 入力セル総数 約2,333個（表紙75 / 計算現状533 / 計算改修後582 / 評価511 / 部材632）。
- **→ セル番地を「捨てる」ことは不可。UIだけ差し替え、内部は番地維持が唯一の正解。**

## 2. ユーザーの不満と原因

### 不満A：「Excel風のセルを使うのは一旦やめたい」
- 単一値項目はすでに `FormField` 化済み。残る“セルっぽさ”の主因は **`RefTable` で出している密な表**（熱損失計算など）と、表の中に混在する **見出し・小計といった「本来1項目1値」のセル**。
- 表として残すべきもの（繰り返し行データ）と、表から出してテキストボックス化すべきもの（基本データ・係数・小計表示）が**同じ表の中に同居**しているのが、分かりにくさの正体。

### 不満B：「見取り図のバックに表が表示され、図の挿入・編集が見えにくい」
**原因（実コードで確認済み）：図面キャンバスの背景が透明で、下のグリッド表が透ける。**
- 編集枠 `.plan-draw-frame`：背景色の指定なし（`globals.css:355-358`）。
- 評価シートのコピー枠：`SheetGrid` のグリッド表の**上に** `SheetOverlays` が `DrawingEditor` を絶対配置（`components/SheetOverlays.tsx`）。
- `DrawingEditor` のSVGは画像があると `background: "transparent"`（`components/DrawingEditor.tsx:207`）。
- → 透明SVG越しに下の `xl-cell`（罫線・背景色）が見える。

## 3. 領域別の推奨方針

| シート／領域 | 現状 | 推奨 | 理由 |
|---|---|---|---|
| 表紙（工事名・住所・診断者 等） | フォーム | **フォーム維持** | 完全に単一値。テキストボックス／セレクトが最適 |
| 計算シート 基本データ（床面積・天井高 等） | フォーム | **フォーム維持＋見直し** | 単一値中心。表に混在する係数も `FormField` に寄せる |
| 計算シート 熱損失計算（開口部/壁/屋根/天井/床/窓/隙間） | RefTable（表） | **表形式を維持** | 部位×項目の繰り返し行。表が最も分かりやすい |
| 表の中の「見出し・小計・合計」セル | 表の中のセル | **表外のテキストボックス／読み取り表示へ分離** | 1項目1値。表に置くと密度が上がり見にくい |
| 部材性能シート（建材マスタ） | RefTable（表） | **表形式を維持** | 多数行の参照表。横スクロール表が適切 |
| 間取り図 | PlanDrawing（図面） | **背景を不透明な“用紙”に**（§4） | 図の挿入・編集を見やすく |
| 評価シート（帳票） | SheetGrid 忠実 | **変更不可** | PDF出力の本体。Excel忠実が要件 |

**原則**：
1. **繰り返し行データ＝表（`RefTable`）**。
2. **単一値＝テキストボックス／セレクト（`FormField`）**。
3. 表の中の見出し・小計・合計など「1項目1値」のセルは、可能な範囲で表の外に出して `FormField`（読み取り専用表示含む）に寄せ、表は純粋なデータ行だけにする。

## 4. 図面背景の修正方針（不満Bの解消）

図面キャンバスに**不透明な「用紙」背景**を与え、下のグリッドを完全に遮断する。3か所すべてで一貫させる：

1. **編集枠（計算シート）** `.plan-draw-frame` に `background: #fff`（用紙色）＋薄い枠線を付与（`globals.css:355`）。
2. **評価シートのコピー枠** `SheetOverlays` の図面ラッパー `div` に `background:"#fff"` を付与（`components/SheetOverlays.tsx`）。
3. **`DrawingEditor` のSVG最下層**に不透明な白 `<rect width height fill="#fff">` を敷く（`components/DrawingEditor.tsx:211` 直後）。これにより `editable`/コピー/PDF すべてで用紙が出る。

- 効果：画像なしのプレースホルダ状態でも「白い用紙に図を貼る」見え方になり、グリッド表は透けない。
- PDF（html2canvas → ReportFrame の SheetGrid 忠実描画）にも自然に反映され、帳票上の図面枠も白地になる。
- 副作用なし：データ層・座標計算・コピー縮小ロジックには触れない。

## 6. 確認結果（2026-06-15・確定）

- **進め方**：Phase 1（図面背景）と Phase 2（表/フォーム整理）を**まとめて実装**する。
- **優先領域**：①計算シートの熱損失計算、②部材性能シート、③間取り図まわり。
- **表の整理方針**：表は**データ行のみ**にし、見出し・小計・合計など1項目1値のセルは表の外へ出して **テキストボックス／読み取り表示** に分離する（＝推奨案を採用）。

## 6.1 実装中に判明した重要事項（熱損失計算の実構造）

実データ（`workbook-model.json`の壁部 r66-121）を精査した結果、熱損失計算セクションは
**「見出し＋データ行＋小計」の単純な表ではなく、計算ワークシートそのもの**だった。

- セルの配置自体が数式構造を表現：`Ｒｔ ＝ Ｒ① ＋ Ｒ② ＋ …`、`＝`・`×`・`＋`の演算子が独立セル、中間計算（`D78=H69`）が散在。
- 1セクション内に「外壁部／内壁部／室内ドア部」など複数の小ブロックが入れ子。
- **ユーザーが実際に入力するのは少数**：建材ドロップダウン（D69〜D74 等）と通気層フラグ、一部の面積のみ。残りはすべて計算結果の表示。

→ よって「見出し・小計を機械的に表の外へ」だけでは不十分。**入力（建材選択）と最終結果（部位ごとの熱損失）を前面に出し、演算子・中間計算の足場を畳む**のが、コンパクト化の本質。

### 推奨する2案
- **案X（軽量・低リスク）**：表は維持しつつ、`RefTable` を改良して「入力セルを強調・演算子や中間足場セルを淡色化」。各セクション冒頭に**入力サマリ（建材選択）と結果（熱損失◯W）を読み取り表示**で添える。番地・行構成は変えないので回帰リスク極小。
- **案Y（本格・高リスク）**：各小ブロック（外壁/内壁/室内ドア…）を `FormField` 群へ分解し、計算ワークシートは「計算の詳細」として畳む。12ブロック×手作業の構成定義が必要で、入力取りこぼし・誤マッピングのリスクあり。1ブロックの試作→確認→展開が前提。

### Phase 2：熱損失計算の分解（不満A）— 完了（案Y）
- 機構を最小拡張：`FormSection` が「フォーカス表（建材選択）＋結果フィールド（熱損失）」を併置できるよう変更（`sheet-parser.ts` / `FormSection.tsx`）。番地・計算・保存形式は無変更。
- `sheet-layout.ts`：両計算シートの熱損失セクションを、Rt/Ut等の中間計算（演算子の足場）を畳み、**建材選択の小表＋部位ごとの熱損失（読み取り）** に再構成。
  - 現状＝16セクション／改修後＝15セクション。総熱損失量を結果フィールドで前面化。
  - 開口部スケジュール・改修後の基本データ・日射取得率（入力なし）は表のまま維持。

### Phase 2.1：建材選択を表→「ラベル＋ドロップダウン」フォーム化（追加フィードバック対応）
ユーザー指摘：Excel表は1列目がタイトル専用で余白が多く見にくい／ラベル＋テキストボックスが分かりやすい。
- 調査：建材行の D/E/F は3つともドロップダウンだが、**計算で参照されるのは D列のみ**（`H=VLOOKUP(D…)`）。E/F列はどの数式からも参照されない**未使用セル**（全シート grep で確認）。
- 対応：`matFieldsSec` ヘルパーを追加し、建材セクションを `reftable` → `fields` に変更。
  - 各建材スロット＝ラベル＋ドロップダウン1つ（例「建材1：[無し▾]」）。未使用の E/F 列は `excludeAddrs` で除外。
  - 面積・部位ごとの熱損失は読み取りフィールドで併記。隣の「Ｒ値」を単位と誤検出する問題も抑止。
  - 外壁部は通気層（1=あり／2=無し）も入力フィールドとして表示。
- 維持：開口部スケジュール／改修後の基本データ／隙間の気密目安／日射取得率は表のまま（本来表が適切）。

### Phase 2.2：残った表（reftable）の横長・空列対策
ユーザー指摘：基本データ表の1列目が空（タイトル専用）で横長／横スクロールせず入力したい／数値セルは小さくてよい。
- **改修後の基本データ(9-16)** は全て現状シートからの自動計算（入力なし）。横長の表をやめ、ラベル＋値の**読み取りフィールド**に変更。断熱改修面積・開口部スケジュール(18-40)のみ表として分離。
- **RefTable 全体改善**（全reftableに適用）：
  - 内容も入力も無い「空の区切り列」を極小幅(5px)に潰す（結合セルに関与する列は保護）。
  - `table-layout: fixed` ＋ `width:100%` ＋ テキスト折り返しで、**コンテナ幅に収め横スクロールを抑制**。列幅は上限104pxでクランプ。
  - 入力欄のフォント・余白を縮小（数値セルを小型化）。
- 適用対象：開口部スケジュール／断熱改修面積／隙間の気密目安／日射取得率／部材性能シート。

### Phase 2.3：全シートのサブエージェント監査と仕上げ
4体のサブエージェントで「表紙／計算現状／計算改修後／部材性能・評価」と「RefTable機構」を並行監査。
- **機構検証**：`table-layout:fixed`＋`width:100%`＋折り返し＋空列潰し（merges保護）で、全reftableが**コンテナ幅に収束し横スクロールしない**ことを確認。
- 発見した実バグ／改善を反映：
  - `.cell-edit/.cell-select` に **`box-sizing: border-box`** を明示（width:100%＋paddingでセルを数pxはみ出すのを解消）。
  - 参照表内 `select` を `min-width:0; max-width:100%` で**細い列でも縮んで収まる**ように。
  - `MAX_W` 104→**160**（fixedレイアウトなので絶対幅は不変・比率のみ。部材性能の建材名など内容列が読みやすく）。
- 各シート判定：表紙＝fields で問題なし／計算シート＝建材はfields・残る表は収束OK／部材性能＝表のまま収束OK／**評価シート＝PDF帳票の本体(SheetGrid 92列)で意図的に横長・変更不可**。
